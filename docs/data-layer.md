# Data layer: files, database, and cache

This project keeps a clear split between **canonical files**, the **database as index + structured cache**, and **client cache** so the public site does not depend on live AI or third-party APIs for facts you already own.

For migrations, S3 secrets, Edge deploy order, and env vars, see **`docs/deployment-checklist.md`**.

## 1. Files (canonical binaries)

**Primary file bucket: AWS S3.** All RaceCard PDFs in production are stored under the `racecards/` prefix in your configured S3 bucket (`AWS_S3_BUCKET`). Admin upload, customer download, and S3→Postgres sync are built around S3 only.

| Store | Contents | Access |
|--------|-----------|--------|
| **AWS S3** (`racecards/…`) | RaceCard **PDFs** — **primary** object store | Presigned **PUT** via `generate-upload-url`; presigned **GET** via `download-racecard`; listing/ingest via `sync-s3-racecards` |
| **Supabase Storage** (`racecards` bucket) | Optional / legacy | May exist from older DB migrations; **not** part of the primary RaceCard path. Do not use for new PDF delivery unless you intentionally fork the pipeline. |

The database **never** embeds PDF bytes. It stores **`file_url`** as the **S3 object key** (same key the SDK uses in `GetObject` / `PutObject`). Public UIs use **`racecards_public`**, which omits `file_url` so anonymous users cannot see object keys.

## 2. Database (source of truth for metadata and business data)

| Area | Tables / views | Role |
|------|----------------|------|
| **Race listings + display cache** | `racecards`, **`racecards_public`**, `metadata` (JSONB), `metadata_updated_at` | One row per published card: pointer to file + **cached** weather, post times, conditions, etc. Updated by admins or your ingest job — **not** re-fetched on every page view. |
| **Assistant reference text** | `site_content` (`key`, `body`, `updated_at`) | Optional DB-backed copy of static assistant knowledge. If a row exists for **`racing_assistant_knowledge`**, the `racing-assistant` Edge function uses it; otherwise it falls back to the bundled `knowledge.ts`. Update via SQL or service role (no anon access). Example: `INSERT INTO site_content (key, body) VALUES ('racing_assistant_knowledge', '...') ON CONFLICT (key) DO UPDATE SET body = EXCLUDED.body, updated_at = now();` |
| **Commerce & users** | `credit_balances`, `credit_packages`, `transactions`, `credit_ledger`, `profiles`, … | Standard relational source of truth. |
| **CMS** | `pages`, `contact_submissions`, … | Content and operational data in Postgres. |

**Ingest pattern (recommended):** your pipeline writes **file → S3**, then **row → `racecards`** (sync or admin), then **`metadata` JSON** on that row (or a follow-up `UPDATE`). The website reads only Postgres + signed file URLs when the user downloads.

**Batch PDFs in `race cards/` (Equibase-style names):** each file is **`XXX` + `YYMMDD`** (three letters for the track, then six digits for the date), optionally **`-N`** for another card the same day (e.g. `Kee260411-1.pdf`). Stray **`^`** characters in some exports are ignored. Run `npm run prepare:racecards` to copy into `racecards-staging/` as `TRACKCODE_YYYY-MM-DD.pdf` (see `scripts/racecard-track-prefixes.json` for letter codes). Upload from **Admin → Racecards**, then **Sync from S3**. Same-day duplicates become `KEE_2026-04-11__2.pdf`, etc.

## 3. Cache (fast repeat reads)

| Layer | Mechanism | Typical use |
|-------|-----------|-------------|
| **Browser / app** | **TanStack React Query** (`staleTime` / `gcTime`) | Racecard listings (`useRacecardsPublicForDate`), credit packages, balances. Reduces duplicate Supabase calls and stabilizes UX. |
| **After admin edits** | `queryClient.invalidateQueries` | e.g. saving race **metadata** invalidates public racecard queries so the homepage and browse page pick up new JSON without a full reload. |
| **CDN / HTTP** | Set on your static host (Vercel, etc.) for `index.html` / assets | Standard static asset caching; API responses are mostly Supabase/Edge (not heavily CDN-cached here). |

**What is not cached as “facts”:** the **racing-assistant** LLM still calls the provider per question; only the **knowledge base text** is loaded from DB (if present) or file bundle, not re-generated by AI each time.

## Summary

- **Files** = **S3 primary bucket** — RaceCard PDFs only; `file_url` in Postgres is always an S3 key for those objects.  
- **Database** = pointers, business data, and **JSON snapshots** (`metadata`, optional `site_content`).  
- **Cache** = React Query (+ invalidation) so repeat views hit memory, not the network every time.
